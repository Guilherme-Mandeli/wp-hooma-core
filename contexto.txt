O Hooma é um plugin de infraestrutura (Framework Plugin) que atuará como um Service Container para as funcionalidades customizadas da empresa.

O objetivo é substituir a prática de códigos soltos no functions.php ou múltiplos "micro-plugins" espalhados, centralizando tudo em uma arquitetura modular, segura e padronizada. O Hooma gerencia, carrega e padroniza a interface, enquanto a lógica de negócio reside em Módulos Independentes.

2. Arquitetura do Sistema
2.1. Isolamento de Diretórios (Sandboxing)
Diferente de plugins comuns, a lógica de negócio não ficará dentro da pasta /plugins/. Isso previne remoção acidental e separa "Core" de "Regra de Negócio".

Plugin Core: wp-content/plugins/hooma/ (Gerenciador)

Regra de Negócio: wp-content/hooma-modules/ (Módulos)

2.2. Filosofia "Thin Core" (Core Magro)
O plugin hooma deve conter o mínimo de código possível. Suas únicas responsabilidades são:

Discovery: Escanear a pasta hooma-modules.

Loading: Ler metadados e carregar módulos ativos.

Standardization: Prover classes base para UI (Header, Tabs) e Autoloading.

3. Especificação dos Módulos
Para o time criar um novo módulo, deve-se seguir a "Regra de Estrutura Interna". Um módulo é tratado como um "plugin dentro do plugin".

Estrutura de Pastas Obrigatória
Plaintext

hooma-modules/
└── nome-do-modulo/
    ├── index.php        (Ponto de entrada único + Metadados)
    ├── includes/        (Classes PHP - PSR-4 Autoloading)
    ├── assets/          (CSS/JS específicos do módulo)
    └── views/           (Arquivos HTML/PHP de renderização)
Cabeçalho de Metadados (index.php)
Todo index.php deve conter um cabeçalho de comentário similar ao do WordPress, com uma flag crítica de performance:

Module Name: Nome legível.

Description: O que faz.

4. Padrões de Desenvolvimento (Coding Standards)
4.1. Autoloading e Namespaces
O Hooma implementa um Autoloader inteligente. O desenvolvedor não deve usar require_once para classes dentro de includes/.

Namespace Raiz: HoomaModules\

Convenção: HoomaModules\{NomeDoModulo}\{Classe} mapeia para hooma-modules/{nome-do-modulo}/includes/{Classe}.php.

Exemplo: A classe HoomaModules\Financeiro\Relatorio será automaticamente buscada em hooma-modules/financeiro/includes/Relatorio.php.

4.2. Idiomas
Código: Variáveis, funções e classes em Inglês.

Comentários: Em Português.

Estilo: Comentários narrativos ("Por que isso foi feito assim?") para lógica complexa. Comentários diretos para trechos óbvios.

5. Interface de Usuário (UI Kit)
Para garantir consistência visual e velocidade de desenvolvimento, os módulos não devem criar sua própria estrutura HTML para o painel administrativo. Devem utilizar a classe estática Hooma_UI.

Componentes Disponíveis:

Header: Título padronizado, subtítulo e breadcrumbs.

Tabs: Navegação interna do módulo (ex: Geral | Configurações | Logs).

Container: Wrapper padrão do WP Admin.

Footer: Copyright e versão.

6. O Painel de Gerenciamento
O Hooma adicionará um menu no WordPress.

Menu Principal: Hooma.

Submenu: Módulos.

A tela "Módulos" simulará a experiência nativa de plugins do WordPress (tabela WP_List_Table), permitindo Ativar e Desativar cada módulo individualmente. Isso permite desligar uma funcionalidade quebrada sem derrubar o site inteiro.
